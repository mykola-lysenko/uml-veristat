# JIT expansion attribution — selftests corpus on UML, bpf-next @ 520d7d794

**Question.** Production dashboards (evpf-viz) show ~30% more native x86
instructions than BPF instructions for the same programs. What drives that
expansion, and is the probe-mem guard fold
(`docs/jit-probe-mem-guard-optimizations.md`, opt #1) the right lever?

**Method.** `scripts/jit_expansion.py` boots 4 sharded UML guests; each loads
every selftests `*.bpf.o` via `bpftool prog loadall` and saves xlated + jited
text dumps. Host-side analysis counts BPF insns (bytes_xlated/8), native insns
(jited disasm lines), probe-mem guard sites (matched on the jited side —
xlated dumps mask `BPF_PROBE_MEM` back to `BPF_MEM` in
`bpf_insn_prepare_dump`), entry-nop padding, and UML-patch call overhead.
`scripts/jit_expansion_model.py` then fits native insns per BPF opcode class
by least squares over all programs (R² = 1.0000, MAE 3.1 insns/program,
intercept ≈ 0).

| source | insns | share of excess |
|---|---|---|

| exit epilogues | 14,665 | 29% |

| unconditional jmp + tail calls | 3,026 | 6% |
| ld_imm64 collapse (credit) | ≈ −12,000 | −24% |

## Interpretation

- **On this corpus the guard is not the dominant term** (0.8% of native
  insns) — selftests are ALU-heavy and mostly use *trusted* pointers, which
  the verifier does **not** convert to `BPF_PROBE_MEM` (only untrusted
  BTF pointers get guards, `kernel/bpf/fixups.c` convert path). Fleet
  composition differs: production LSM/tracing programs walk untrusted BTF
  pointers heavily (fbk5 snapshot: 1,283 sites, ~51 KB guard code).
- **The guard-heavy tail here looks exactly like production**: dump_tcp6
  48%, dump_tcp4 42%, connect_unix/sendmsg_unix 55%, test_map_kptr_ref 27%
  of native insns are guard code. For such programs opt #1 (−2 insns/site of
  9) and the deferred opt #2 (−1 more) are the right levers.
- **The gap to the fleet's ~30%** is plausibly: guard-heavy program mix,
  more/larger subprog epilogues, and whether the dashboard counts entry-nop
  padding as instructions. Corpus-independent terms (cmp+jcc ≈ 2×
  conditional-jump count, epilogue ≈ 7×exit, ~15 nops/prog) can be computed
  directly from any fleet program's xlated dump for comparison.
- **UML caveat now quantified**: our syscall-wrapper patches add
  pushq/popq %r9 around every helper call plus a %gs params frame and
  indirect movabsq+callq (native x86 emits one direct call). That is 10.4%
  of native insns here and must be subtracted before comparing UML ratios
  with real hosts (the harness reports `ratio_native_equivalent`).

## Guard opt #1 — measured (tasks #32/#33)

The fold (`patches/uml-veristat/0009c-x86-bpf-fold-probe-mem-low-bound-
into-guard-offset.patch`, same diff as the doc) was built into the stack and
the full harness re-run on the patched kernel. A/B against the baseline run
(`scripts/jit_expansion_diff.py`, joined on the same 1,938 programs):

| metric | baseline | opt #1 | delta |
|---|---|---|---|
| jited insns | 514,779 | 513,851 | **−928** |
| jited bytes | 2,004,979 | 2,000,870 | **−4,109** |
| guard sites | 488 | 488 | 0 |
| guard insns | 4,344 | 3,416 | −928 |

Every program's whole-body insn delta equals its guard-prefix delta — the
fold provably touched nothing but guard prefixes. Savings: 1.90 insns and
8.4 B per site (2 insns for `off≠0` sites, 1 for `off==0`; doc's "up to 2 /
10 B" bound holds). Guard cost drops 4,344 → 3,416 insns (−21%).

Correctness on the patched kernel: `check_expectations` green (2,976
verified, identical to baseline), `check_arena_expectations` green (58/2),
and the PROBE_MEM-exercising test_progs subset (jit_probe_mem, ksyms_btf,
map_kptr, rcu_read_lock, btf_dump) passes. The three failing tests in that
subset (iters/task_vma, iters/css_task, task_kfunc/test_task_from_vpid_*,
map_kptr_race) were A/B'd against a `--skip-patches=0009c` build: **identical
subtest failures without the fold** — pre-existing UML environment issues,
not regressions. Verdict: opt #1 is correctness-clean and size-exact on UML;
remaining validation (wall-clock, native-x86 build) belongs on real
hardware.

## Runtime microbenchmark (benchmarks/probe-mem/)

The katran-style selftests bench (`bench_xdp_lb`) cannot exercise this
change — its program JITs byte-identically on both kernels (1,294 insns,
0 guard sites; XDP code never derefs untrusted BTF pointers). Instead,
`benchmarks/probe-mem/probe_mem_bench.bpf.c` is a guard-saturated XDP
program: 32 unrolled `bpf_rdonly_cast`s over our own 2 KB map value with
6 field loads each — 192 guard sites, all in-span (guards always pass, no
exception fixups), driven by `bpftool prog run ... repeat 2000000`, in-kernel
duration averaging, 7 trials. A `plain_bench` control does the same loads
through the trusted map pointer (no guards) to pin the noise floor.

| | stock guard | 0009c folded | delta |
|---|---|---|---|
| guard_bench jited insns | 2,565 | 2,213 | −13.7% |
| guard insns (192 sites) | 1,696 | 1,344 | −20.8% |
| guard_bench ns/run (median of 7) | 134 | 93 | **−30.6%** |
| plain_bench ns/run (control) | 44 | 43.5 | ~0 ✓ |

Static counts match theory exactly (stock 160×9 + 32×8; folded 192×7).
The runtime gain exceeding the insn-count reduction is plausible
micro-architecturally (the removed movabsq is a 10-byte decode plus a
dependency in the address-check chain), but per-insn IPC effects are
host-CPU-specific. Caveats: UML guest = single host process, host
CLOCK_MONOTONIC, no PMU; the flat control and 2M-rep averaging make the
relative signal credible, but fleet numbers must come from real hardware.
Production effect scales with dynamic guard density: this microbench is
~76% guard insns; a 48%-guard program like dump_tcp6 should see roughly
half this gain on its guarded paths, and guard-free programs (katran)
exactly zero.
