============================

Current findings
----------------
1. build.sh used make -k, but not the selftests permissive mode. That allows
   make to continue after compile failures, but it does not guarantee a linked
   test_progs binary. Passing BPF_STRICT_BUILD=0 enables the selftests
   skip-on-failure rules and links a partial test_progs for runtime triage.
2. A focused rebuild with BPF_STRICT_BUILD=0 produced:
   uml-veristat/.build/selftests-output/test_progs
   uml-veristat/.build/selftests-output/veristat
3. test_progs --help runs on the host, confirming the binary is linked.
4. One local experiment changed bpf_iter_task_btf.c to include btf_ptr.h. That
   fixes an incomplete struct btf_ptr compile failure when UML vmlinux.h only
   has a forward declaration. This looks like a small generic selftest cleanup,
   but it is not required for producing a permissive test_progs binary.
5. test_progs --list booted under UML but exited 255 before printing tests when
   CONFIG_ASYMMETRIC_KEY_TYPE was unset. test_progs unconditionally registers
   an asymmetric session key at startup, so the UML config needs the asymmetric
   key type plus X.509/PKCS#7 parser support before runtime selftests can run.
6. After enabling the asymmetric key type and parsers, add_key() reached the
   key type but still failed with ENOPKG. CONFIG_CRYPTO_RSA is present and
   /proc/crypto exposes rsa and sha256, but pkcs1(rsa,sha256) is still not
   instantiated for the selftest certificate. For now, keep test_progs usable by
   warning instead of aborting when the startup key registration fails.
7. test_progs expects bpf_testmod.ko in its current working directory and
   handles unloading/reloading it itself. The UML runner should make module
   files visible from the selftests output directory rather than pre-insmoding
   bpf_testmod before test_progs starts.
8. The local LLVM install has working clang/llc, but optional LLVM disassembler
   linking for test_progs fails because llvm-config references missing
   libLLVM-23git.so metadata. build.sh should pass SKIP_LLVM=1 for selftests so
   optional JIT disassembly support does not block linking test_progs.
9. The focused arena/runtime smoke set now passes under UML:
   arena_htab, arena_list, arena_strsearch, arena_atomics, arena_direct_value,
   and queue_stack_map all pass. arena_spin_lock still skips.
10. A monolithic full test_progs run is not a good debugging unit yet. It hit a
    host-side timeout before producing a useful baseline. The runner now
    line-buffers test_progs output and dumps any partial guest output on
    termination; validate that with chunked runs rather than another blind full
    run.
11. fentry/fexit-style tests against the synthetic __x64_sys_* wrappers need
    three UML runtime fixes: patchable 5-byte wrapper entries, a UML text-poke
    path that can write protected host mappings, and UML guest memory below the
    x86 signed 32-bit direct-call limit. With UML_MEM=1792M,
    bloom_filter_map now passes; with 2048M, BPF trampoline allocation lands
    just beyond the call range from UML kernel text.
12. The synthetic __x64_sys_* wrappers also need to be in the actual UML
    syscall dispatch path. Without that, fentry/fexit links can attach to the
    wrappers but never execute. Routing the selected syscall numbers through the
    wrappers makes map_btf and user_ringbuf pass.
13. Enabling FTRACE_SYSCALLS and mounting tracefs creates
    /sys/kernel/tracing/events/syscalls/* directories, but the event id files
    remain absent because UML lacks PERF_EVENTS. Tracepoint auto-attach still
    fails while reading the perf event ID, and raw tracepoint attach remains
    blocked because CONFIG_BPF_EVENTS depends on PERF_EVENTS.
14. Light-skeleton loader failures for ringbuf/map_ptr were caused by
    BPF_PROG_TYPE_SYSCALL loader programs calling bpf_probe_read_kernel()
    while UML builds without BPF_EVENTS. Adding probe-read helper
    implementations to the UML verification stubs makes all ringbuf subtests
    pass. map_ptr now loads and runs, and its remaining failure is direct
    struct bpf_ringbuf CO-RE field-offset access: bpf_ringbuf_query() reports
    producer_pos=136, but direct rb->producer_pos still reads zero.
15. On distros whose compiler defaults to -fcf-protection=full (Ubuntu,
    Fedora), the UML kernel was built with a 4-byte endbr64 before every
    function because UML makefiles, unlike native arch/x86, never pass
    -fcf-protection=none. The endbr64 displaces the synthetic wrappers'
    patchable_function_entry slot, so BPF trampoline attach failed with
    -EBUSY on such hosts while working on hosts without the CET default.
    Patch 0013b builds UML with -fcf-protection=none; map_btf, user_ringbuf,
    and the map_in_map fentry.s paths recover on Ubuntu 24.04 hosts.
16. verifier_arena/basic_alloc1 panicked the UML guest because segv() had no
    path for BPF exception fixups: kernel-mode faults on user-range addresses
    panicked before any fixup attempt, and the legacy arch_fixup() misparses
    modern typed extable entries. In addition, fixups edit the uml_pt_regs
    copy while the faulting context resumes from the mcontext, so the fix
    must write the registers back or the instruction faults forever. Patch
    0016 adds bpf_extable_fixup() (dispatching to ex_handler_bpf()) plus
    mc_set_regs(), and verifier_arena now runs to completion: deliberate
    use-after-free arena reads return zero as expected.
17. With the panic fixed, remaining arena failures are the kmalloc_nolock()
    class: *_nosleep subtests fail outright, and sleepable basic_alloc1
    fails at page re-allocation after free (retval 8), pointing at the
    free-side range_tree_set() node allocation. arena_kfuncs_under_bpf_lock
    also fails. verifier_arena_large hangs the guest and needs a runner
    denylist entry (isolated repro: no output, killed by timeout).
18. Rechecked on the 2026-07-10 bpf-next tip: map_ptr passes, so the direct
    struct bpf_ringbuf CO-RE field access issue (open question 4) appears
    fixed by upstream drift. cgroup net_cls setup failures were a kernel
    config gap; build.sh now enables CGROUP_NET_CLASSID and CGROUP_NET_PRIO,
    after which the cgrp1_* environment works. Remaining cgrp_local_storage
    failures are tp_btf (-3, BPF_EVENTS gap) and skel loads with -22 in
    sleepable/RCU paths.
19. kmalloc_nolock() failed on UML because boot_cpu_data.x86_capability is
    never populated: system_has_cmpxchg128() keys on X86_FEATURE_CX16, so
    slab caches never got __CMPXCHG_DOUBLE and kmalloc_nolock() returned
    NULL unconditionally (mm/slub.c requires freelist ABA support). UML also
    lacked HAVE_ALIGNED_STRUCT_PAGE, which compiles the ABA support out
    entirely. Patch 0017 probes host CPUID leaf 1 in setup_arch() and
    selects HAVE_ALIGNED_STRUCT_PAGE; with it, all 20 verifier_arena
    subtests pass (including *_nosleep and re-allocation after free) with
    no regressions in the other chunks. The verifier_arena_large hang is
    unaffected and has a separate cause.
20. Patches 0006/0006b (arena preallocation) removed from the stack: with
    patch 0017 in place, the full verifier_arena suite, arena_smoke, the
    veristat arena corpus, and the top-level corpus all pass without them.
    They were workarounds for the kmalloc_nolock() failure fixed at the
    root. The stack shrinks by two kernel patches.

Runner requirements
-------------------
1. Reuse the installed uml-veristat kernel and modules where possible.
2. Boot with hostfs=/ so build output paths remain visible inside UML.
3. Mount proc, sysfs, devtmpfs, bpffs, cgroup2, and debugfs where available.
4. Bring loopback up before network-adjacent tests.
5. Symlink bpf_testmod.ko, bpf_test_modorder_x.ko, and bpf_test_modorder_y.ko into the selftests output directory before test execution.
6. Keep skip/block lists explicit and narrow; start with known hang/crash candidates only after repro.

   kmalloc_nolock() issue (persists after patch 0017) and produces no
   output before the host-side timeout.

Chunked execution plan
----------------------
1. Start with short, high-signal smoke chunks:
   arena_htab, arena_list, arena_strsearch, arena_atomics, arena_direct_value,
   arena_spin_lock, queue_stack_map.
2. Run verifier and CO-RE chunks separately so verifier regressions do not get
   mixed with runtime environment failures.
3. Run map and cgroup chunks separately because they are most relevant to the
   current veristat map-definition fixes.
4. Save every chunk log under uml-veristat/.build/test-logs/ with the selected
   test names in the filename.
5. Only after the chunks are understood, try a full run with a generous timeout
   and treat it as a coverage baseline, not as the first debugging signal.
