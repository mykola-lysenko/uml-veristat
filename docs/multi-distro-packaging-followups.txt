------------------

GLIBCXX_3.4.29) cannot run the prebuilt LLVM release binaries (LLVM 22
needs GLIBCXX_3.4.30). EL9 users can still build locally with
./build.sh --llvm-source.

Runtime dependencies by distro (run 28605918837, 2026-07-02)
------------------------------------------------------------
linux (UML kernel):
- ubuntu-24.04, ubuntu-26.04, fedora-latest, arch-latest: libc only
  (linux-vdso, libc.so.6, ld-linux-x86-64.so.2).
- rocky-10: "not a dynamic executable" — fully static. Worth
  investigating why the EL10 toolchain links UML statically before
  relying on it either way.

veristat:
- All distros: libelf.so.1, libz.so.1, libzstd.so.1, libc.so.6.
- arch-latest additionally links libpcap.so.1, libnl-genl-3.so.200,
  libnl-3.so.200, libdbus-1.so.3, libsystemd.so.0, libgcc_s.so.1.
  The selftests Makefile feature-detection picks these up because the
  arch base image ships the corresponding libraries. This makes the
  arch tarball less portable than the others; if a lean, uniform
  dependency set is wanted, pin the feature set at build time (or
  build veristat static, see open question 2).

Implication for packaging: native package dependencies are
libelf + zlib + libzstd everywhere (plus the arch extras above);
the UML kernel needs nothing beyond glibc.

Packaging follow-ups
--------------------
1. Add an explicit package staging directory to build.sh so tarballs and native packages use the same file list.
2. Teach the wrapper to look in a system install directory, likely /usr/lib/uml-veristat, before falling back to ~/.local/share/uml-veristat.
3. Add nfpm metadata for .deb and .rpm packages from the staged tree.
4. Add install-and-smoke-test jobs for generated .deb and .rpm packages in clean distro containers.
5. Decide whether Arch should stay tarball-only or get a PKGBUILD/native package path.
6. Add release automation for tagged builds: checksums, provenance, and package artifacts attached to a GitHub release.

Open questions
--------------
1. Should packages bundle libelf/libzstd dependencies or declare distro-native dependencies?
2. Do we want fully static veristat builds to reduce distro compatibility risk?
3. Should nightly package builds track bpf-next master, while release package builds pin an explicit bpf-next commit?
